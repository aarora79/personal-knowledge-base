# Vibe Coding

Vibe coding is a philosophy of software creation where the builder focuses on the desired outcome and aesthetic rather than the underlying code implementation. Instead of writing code line by line, the developer describes what they want -- the "vibe" -- and an AI assistant generates the technical implementation. The term captures a shift from traditional software engineering, where understanding every line of code is paramount, to a more outcome-driven creative process.

In the context of AI-assisted development, vibe coding enables people who are not traditional software engineers to build functional, polished software artifacts. The frontend-slides project exemplifies this: it enables non-designers to create animation-rich HTML presentations by reacting to visual previews rather than writing CSS or JavaScript. The user picks the style they like from generated options rather than articulating design preferences in technical language.

The philosophy rests on several beliefs: that you do not need to be a designer to make beautiful things (you just need to react to what you see), that dependencies are debt (a single HTML file outlasts a framework-dependent project), and that every presentation should feel custom-crafted rather than template-generated.

Vibe coding stands in interesting tension with Peter Naur's theory of programming as theory building, which argues that true programming knowledge is the mental model held by the programmer, not the code artifact. Vibe coding shifts who holds the theory -- the AI holds the implementation theory while the human holds the intent theory.

## Key Points

- Vibe coding prioritizes outcome and aesthetic over implementation details
- Enabled by AI tools that generate code from high-level descriptions
- Lowers the barrier to creating polished software for non-engineers
- Emphasizes "show, don't tell" -- users react to visual options rather than describing preferences in words
- Represents a shift from code-centric to intent-centric software creation
- Challenges traditional assumptions about who can build software
