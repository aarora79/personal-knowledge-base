# Verification Pipelines for Code Evolution

Verification pipelines for code evolution are multi-stage correctness checking systems that ensure autonomously evolved code remains functionally correct throughout the evolution process. In the context of SATLUTION, the verification pipeline was described as "indispensable" for repository-scale solver evolution, preventing the evolutionary process from veering into unsound territory.

## Two-Stage Verification Architecture

SATLUTION employs a stringent two-stage verification pipeline that every iteration must pass before receiving performance feedback.

**Stage 1: Compilation and Smoke Testing** - After the Coding Agent produces a new code version, the system attempts compilation with standard settings. Any compile error immediately flags the iteration as failed, with feedback sent to the agent to fix the issue. If compilation succeeds, the solver runs on 115 trivial CNF formulas from the Kissat test repository, exercising basic functionality (unit propagation, simple satisfiable and unsatisfiable cases). The system monitors for runtime errors including crashes (segmentation faults, assertion failures) and incorrect answers. Failed iterations are halted with specific diagnostic information (e.g., "segmentation fault in preprocessing").

**Stage 2: Full Correctness Validation** - Only solvers passing Stage 1 proceed to this stage. The solver is tested on a larger set of benchmark instances with known outcomes from past competition problems. For each instance solved: if the solver reports "SAT," the returned variable assignment is verified against the formula; if it reports "UNSAT," the produced DRAT (Deletion Resolution Asymmetric Tautology) proof is validated using an external proof checker. Any deviation (wrong answer or invalid proof) triggers a failure.

## Design Rationale

Early experiments without the verification pipeline relied on end-of-benchmark validation only, which wasted at least 1.5 hours per cycle when errors went undetected until full evaluation. The two-stage approach provides immediate, fine-grained signals that prevent wasted evaluations and preserve the integrity of the evolution's reward signal. The fast shallow tests (Stage 1) catch the majority of issues cheaply, while deep validation (Stage 2) ensures formal correctness.

Throughout SATLUTION's experiments, no solver version that passed both stages ever produced a misclassification on competition benchmarks. Across the full evolution process, 11 solver variants exhibited partial segmentation faults, 4 failed functional or proof validation, and 9 showed significant performance regressions. All were automatically detected and pruned.

The Stage 2 feedback flow was manually crafted by the authors rather than generated by agents, reflecting the critical importance of verification quality in the evolution loop.

## Key Points
- Two-stage pipeline: fast shallow tests (Stage 1) followed by deep formal validation (Stage 2)
- Stage 1 catches compilation errors, crashes, and obvious incorrect results on 115 test cases
- Stage 2 validates SAT assignments against formulas and UNSAT proofs via external DRAT checker
- No solver passing both stages ever produced a misclassification on competition benchmarks
- Without the pipeline, 1.5+ hours wasted per cycle on undetected errors
- Verification feedback quality is critical; Stage 2 feedback was manually crafted by authors
